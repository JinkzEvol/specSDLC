An explainer for engineering teams · v1.0

How a portable spec, a roster of narrow agents, and a strict workflow combine to deliver software autonomously — without losing the plot.

Spec SDLC is not a single AI that "writes code." It is a portable workflow factory: a typed spec defines the truth, prompts and commands route work, narrow agents do the work, gates check the work, and a memory layer keeps yesterday's lessons attached to tomorrow's decisions. The whole kit is designed to be transplanted into any repository and regrow repo-specific behaviour.
"The orchestrator decides the route. Spec creators define it. Stage operatives walk it. Domain leads confirm it is safe. Memory preserves what the route taught you."

§01 · Mental model

Three artefacts. Four agent layers. One loop.

Hold this image and the rest of the document is just elaboration. Spec SDLC has three durable artefacts that change shape over time, four cooperating agent layers that act on those artefacts, and one continuous learning loop that makes the system better at acting. Crucially, the kit is designed to be transplanted — copied into a new repo, profiled against that repo's environment, and put to work without dragging along the assumptions of its source.

Spec → Agents → State → back to Spec. The dashed arc is what makes this a system, not a workflow.
§02 · The argument

Why a system, instead of just "an AI coder"?

TL;DR

One generalist agent forgets, drifts, and silently fails. A roster of narrow agents bound to a typed spec, governed by an orchestrator and audited by gates, produces evidence — not vibes. And because spec SDLC is portable, the same discipline travels with the kit into every repo it lands in.

The failure modes a system prevents

  • Code outruns the spec; no one notices
  • Tests skipped to "make it pass"
  • Two parallel agents stomp on the same file
  • Status docs become fiction
  • One incident, then the same incident again next month
  • Lessons trapped in chat history instead of repo memory
  • Generic best-practice copied without regrowing repo specifics

The discipline this system imposes

  • Specs lead, code follows
  • Every gate decision is evidence-backed
  • One primary lane per work packet
  • Status files are version-controlled truth
  • Lessons promote into memory, knowledge, decisions, or rules
  • The orchestrator is the default, not a last resort
  • Environment profile re-grows assumptions on transplant
§03 · The spec

The spec is the constitution. Everything else is enforcement.

The spec is not a "design doc that the team reads once." It is the operating contract the agents read on every run. Every functional requirement maps to a contract test or eval; every gate decision is anchored in spec phases; every shared type is defined once and imported everywhere.

In a transplanted kit, the spec is whatever the host repo declares it to be — a product requirements doc, an architecture book, a phase plan, a build constitution, or a combination — and the spec hierarchy is recorded in docs/ENVIRONMENT_PROFILE.md so every agent reads from the same source of truth.

The reading rule

Before the orchestrator does anything, it reads the spec hierarchy. A request that contradicts the spec doesn't get implemented — it raises a variance that updates the spec first.

Critical distinction

Stage operatives issue verdicts. Domain leads gather evidence. They are not interchangeable. If Security Lead says "ship it" and Quality Gate says "fail," Quality Gate wins. The lead's job was to make sure the gate had what it needed.

§06 · Agent catalog

All 22 agents, by what they actually do.

SDLC Orchestrator
Control plane · default entry
+

The control plane. Reads SYSTEM_STATE.md, the spec hierarchy, and the drift report; chooses the next agent; enforces stage order; routes durable lessons.

Use when: work spans multiple stages, ownership is unclear, or you don't know where to start. Default routing: unclear blast radius → Change Impact; planning only → Planner; pre-code → Spec Guardian; implementation → TDD Guide.
Build Lead
Control plane · go/no-go
+

Owns gate-readiness evidence and the go/no-go call. Coordinates across workstreams when several lanes need to ship together.

Strict rule: never approves on narrative — always demands tests, state, and domain evidence. Routes multi-stage work back to the SDLC Orchestrator.
Change Impact
Spec creator · blast radius
+

Maps what a planned change touches before any code is written. Identifies primary targets, dependents, affected tests, schema impact, deployment impact, and required reviewers.

Use when: scope is non-trivial. Does not: design implementation. Hands off to Planner with a risk map.
Planner
Spec creator · plan
+

Implementation-planning specialist. Produces objective, spec references, Critical Control Points, deliverables, assumptions, risks, verification plan, and recommended first slice.

Use when: the request is planning-only. Routes back to Orchestrator if work extends into implementation, deployment, or incident follow-through.
Spec Guardian
Spec creator · compliance
+

The fiercest gatekeeper. Extracts testable requirements from the spec, compares plan/implementation against them, blocks contradictions, warns on gaps. Recommends spec updates rather than approving violating code.

Use: before implementation, during audits. The reason functional requirements survive contact with code.
Architect
Spec creator · design
+

Reviews system design when a change crosses boundaries, stresses platform limits, or introduces new structure. Produces options, trade-offs, boundary impacts, platform-fit risks, recommendation with rationale.

Use when: a new ADR might be needed or an existing one is being challenged.
Database Reviewer
Spec creator · data
+

Reviews schema, migrations, query contracts, persistence boundaries, data integrity. Flags uncertain migration scope as blocking.

Use when: changes touch any persisted schema, row-level security policies, or query patterns recorded in the environment profile.
TDD Guide
Stage operative · implementation
+

Enforces RED → GREEN → REFACTOR around the repo's real toolchain. Identifies critical contracts, writes failing tests first, implements minimally, refactors safely.

Strict rule: uses commands defined in ENVIRONMENT_PROFILE.md; if missing, stops and requests profiling rather than guessing.
Code Reviewer
Stage operative · review
+

Reviews for quality, maintainability, correctness risk. Reports findings as BLOCKING / HIGH / MEDIUM. Skips low-confidence noise.

Use: after implementation, before security review.
Security Reviewer
Stage operative · review
+

Reviews for security risk: secrets, auth, injection, unsafe input, sensitive logging, boundary isolation, deployment-time exposure. Blocks on serious issues.

Use: on any change touching auth, secrets, external integrations, or tenant boundaries.
Verify
Stage operative · verification
+

Stage 7 verification gate. Runs the repo's real build/typecheck, tests, and runtime/packaging/wiring checks. Diagnoses and reports — does not fix.

Verdicts: PASS / FAIL / UNVERIFIED. Missing evidence = UNVERIFIED, treated as blocked.
Quality Gate
Stage operative · binary gate
+

Stage 8 binary gate after Verify. Reads spec, review outputs, security findings, and quality/criteria.md. Pass or fail — no middle.

Hard rule: missing evidence is fail; Verify must pass first; spec contradictions block advancement.
Phase Implementation Assess
Stage operative · phase gate
+

The phase-transition gatekeeper. Conservative by design. Allowed statuses: NOT STARTED / PARTIALLY IMPLEMENTED / IMPLEMENTED BUT NOT VERIFIED / VERIFIED BUT NOT TRANSITION READY / TRANSITION READY.

Use: at every phase exit. Caps readiness conservatively when any hard gate fails.
Regression Sentinel
Stage operative · regression
+

Protects the baseline after changes land. Checks build/typecheck regressions, test count or pass-rate regression, missing test updates, coverage regression, contract or golden-baseline drift.

Reads: baselines from ENVIRONMENT_PROFILE.md and SYSTEM_STATE.md.
Deployment Agent
Stage operative · deploy
+

Two modes: readiness (assess only) and execute (controlled rollout). Normalises environment and boundary, verifies config/auth/secrets/migrations/order, applies only safe reversible fixes when intent is explicit.

Hard rule: never deploys beyond requested scope. Finishes with a hard verdict.
Platform Lead
Domain lead · platform fit
+

Owns runtime constraints, deployment configs, environment parity, migrations and rollout ordering, queue/worker/service topology, infra blockers that could invalidate verify or deploy.

Treats as blocking: config drift, ambiguous deployment boundaries.
Security Lead
Domain lead · security posture
+

Owns auth, authorization, secret handling, data exposure risk, isolation/tenancy guarantees, rollout security posture. Prepares evidence for the Security Reviewer's verdict.

Hard rule: missing proof for sensitive boundaries is blocking, even if the change "looks fine."
Quality Lead
Domain lead · evidence quality
+

Decides whether the evidence packet is good enough for a gate to trust. Owns acceptance criteria, test/contract coverage, baseline protection, soak/readiness evidence, packet completeness.

Hard rule: weak evidence is a blocker even when the narrative sounds confident.
Observability Lead
Domain lead · runtime evidence
+

Proves what happened and whether the system can observe future failures. Logging/trace coverage, metric/alert readiness, error-code discipline, runtime verification, soak-window proof.

Hard rule: if a failure mode cannot be observed, the change is not ready.
Learn
Learning · post-deploy
+

Converts completed work into reusable learning. Reads system state, knowledge index, criteria, recent memories. Writes to knowledge/, decisions/, and (when explicitly approved) quality/criteria.md.

Hard rule: does not invent rules from one-off noise — patterns must be confirmed before promotion.
Postmortem
Learning · incident
+

Turns a failure or rough session into a durable artifact. Distinguishes trigger / visible symptom / direct cause / systemic cause / verification failure / what was fixed / what remains unverified.

Use after: incidents, failed sessions, repeated blockers. Writes to memories/.
Process Engineer
Learning · process mining
+

Reasons from logs, traces, events, metrics, and workflow evidence rather than assumptions. Modes: baseline establishment, deviation detection, troubleshooting, improvement recommendations.

Promotes: proven recurring patterns into memory, instructions, or agents.
§07 · Prompts

Prompts: structured workflow entry points.

Prompts are the standardised recipes. Where a command says "start the orchestrator," a prompt says "do this multi-step recipe with these inputs and these outputs." They live in .github/prompts/ and exist to keep recurring workflows consistent across runs and across repos.

Intake & planning prompts

  • receive-feature-signal — structured intake before spec sync; checks blockers, phase fit, compatibility
  • sync-external-spec — drafts spec updates first, then runs Spec Guardian, then hands to plan-feature
  • plan-feature — runs change-impact, planner, spec-guardian in sequence; produces plan

Verify & deploy prompts

  • verify-phase — build/typecheck → tests → spec compliance → regression → phase assess → one verdict
  • prepare-for-deployment — runs Deployment Agent in readiness mode; normalises env, scope, rollout, constraints
  • execute-deployment — runs Deployment Agent in execute mode; readiness checks first, scoped, hard verdict
  • broadcast-state — produces stakeholder-ready broadcast and refreshes SYSTEM_STATE.md
§08 · Commands

Commands: the user-facing hooks.

Commands live in .claude/commands/ and are how a human (or another agent) invokes the system. Keep command intent narrow — multi-stage requests route through /orchestrate instead of overloading individual commands.

/orchestrate

Default for multi-stage work, incidents, releases, unclear ownership. Reads state, picks entry point, sequences agents.

/plan

Planning-only. Produces objectives, CCPs, assumptions, risks. Defers to /orchestrate if scope expands.

/tdd

Enforces RED → GREEN → REFACTOR for risky or behaviour-changing work. Critical contracts first.

/verify

Runs the real build/typecheck, tests, and routing/packaging checks via commands from ENVIRONMENT_PROFILE.md.

/quality-gate

Binary pass/fail after Verify. Uses spec, review outputs, security findings, and quality/criteria.md.

/checkpoint

Creates clean evidence-backed save points in long-running work.

/learn

Extracts reusable patterns from completed work and incidents into memory, knowledge, decisions, quality.

§09 · Instructions

Instructions: the standards layer.

Instructions are the house rules. They sit in .github/instructions/ and apply across agents and prompts. They are the layer that prevents a bright idea from one agent contradicting an existing rule someone already wrote down.

i
sdlc-workflowDefines the canonical pipeline (External signal → spec sync → change impact → planning → spec guardian → TDD → review → security review → verify → quality gate → phase assess → regression → deploy → broadcast → learn) and the default-bias rule: when in doubt, start with the orchestrator.
ii
environment-regrowthHow transplanted assets regain repo-specific behaviour: read ENVIRONMENT_PROFILE.md, then SYSTEM_STATE.md, then root README, package manifests, CI files, infra config, and tests. Prefer evidence over inherited assumptions. The signature instruction of the kit — without it, the package stays generic.
iii
coding-standardsKeep files focused, prefer explicit types at boundaries, avoid mutation, handle errors explicitly, validate external input, structured logging, simple testable functions.
iv
testing-standardsTests before or alongside risky implementation; preserve baseline; cover failure paths not just happy paths; use commands from ENVIRONMENT_PROFILE.md.
v
security-standardsNo hardcoded secrets, validate untrusted input, review auth at boundaries, check injection risks for the live stack, isolation/tenancy/permission drift is blocking.
vi
deployment-checklistNormalise scope first, real deploy commands known, secrets/auth clear, config parity verified, migrations scoped, verification matches service type, safe-fix policy.

§12 · Lanes

Lanes: strict ownership, parallel-safe.

Spec SDLC partitions the host repository by ownership lanes — not folder convenience. Each lane has primary paths it owns, read-only paths it may consume, and forbidden paths. This is what makes parallel agent work tractable: two agents working in different lanes physically cannot collide.

The kit itself does not prescribe a fixed lane list — your repo declares its lanes during transplant, often in a partitioning document derived from this template. The example below shows a real ten-lane partition used in a TypeScript monorepo with a UI, a Worker BFF, an MCP layer, an orchestrator service, and a database schema. Use it as a model, not a mandate.

The cardinal rule

One primary lane per task. Cross-lane edits are prohibited unless the packet is explicitly marked integration_packet: true and human-approved.

Contract-flow ordering

Cross-lane work moves in a specific sequence. Skip a step and the parallel-safety guarantee breaks.

Adding a new content block: contracts defines schema → producer emits it → UI renders it → integration wires end-to-end → verify checks the round trip.
Dependency budget

A packet may add at most one new runtime dependency. The packet must record purpose, exact pinned version, alternatives considered, security impact, affected deployable, and a removal plan.

Step 04 · Change impact
Map the blast radius before any code.
AgentChange Impact Outputimpact map

Change Impact identifies which lanes are touched, what contracts move, what tests will need updating, and what coordination files might be in play. It deliberately does not design implementation.

Impact map
  • Lanes touched: LANE-CONTRACTS (one new schema), LANE-BFF (one new endpoint), LANE-UI (renderer), LANE-EVAL (one acceptance scenario), LANE-SPEC (status update)
  • Contracts changed: one additive schema
  • Coordination files: package.json (new internal dep on contracts package only)
  • Risk level: medium — many lanes, no auth or schema changes
  • Recommendation: decompose into five lane-owned packets, contract-first
Step 05 · Plan

The Planner produces a packet sequence. Each packet has a primary lane and explicit allowed/forbidden paths.

Checks performed

  • Each packet declares one primary lane ✓
  • Each packet declares allowed paths, forbidden paths, contracts, coordination files, stop conditions ✓
  • No packet modifies auth + integrations + UI together ✓

§15 · Self-improvement

Controlled self-improvement, not autonomous self-mutation.

The system gets better over time — but never by editing itself in production. Every lesson takes a deliberate path: trace → Learn agent → candidate update → human-approved PR → regression check → merge.

Where lessons land

A confirmed lesson goes to exactly one durable home — chosen for the smallest-file-that-truly-owns-it.

Volatile / specific

  • memories/ — incident lessons, what failed, what was fixed, what's still unverified
  • docs/RISK_REGISTER.md — active risks that need watching
  • examples/ — new Gherkin acceptance tests
  • apps/eval-runner/ — new eval cases

Durable / universal

  • knowledge/ — reusable implementation patterns
  • decisions/ — architecture decisions (ADRs)
  • quality/criteria.md — recurring failure patterns promoted to gates
  • The spec itself — when a requirement actually changes
Discipline

Don't promote one-off mistakes into global workflow law. A lesson is durable when the evidence shows the same pattern twice. Otherwise it stays in the run report.

§16 · Quick reference

Glossary & cheat sheet.

Spec SDLC
The portable, lean spec-driven delivery kit described in this document. Designed to be transplanted into any repo and regrow specifics.
SDLC Orchestrator
The control-plane agent. Default entry for any multi-stage work.
Build packet
The structured task contract. Declares lane, paths, contracts, tests, stop conditions.
Lane
A primary ownership zone in the repo. One primary lane per task.
Integration packet
Cross-lane packet, requires explicit human approval. Rare.
Stop condition
A trigger to stop and emit a variance note instead of editing code.
Packet variance
A note saying "this task can't proceed as scoped, here's why" — not a failure.
Functional requirement
A testable requirement extracted from the spec. Each must map to a contract test or eval.
Contract test
A test that pins a shared boundary (schema, type, protocol) so producers and consumers cannot drift.
Eval
End-to-end behavioural test for the agent path or feature scenario.
Walking skeleton
Minimal end-to-end deployable that proves the runtime path before features.
Environment regrowth
The kit's signature instruction: read the host repo and write ENVIRONMENT_PROFILE.md instead of guessing.
P0 drift
Spec/code/test/doc mismatch severe enough to block release.
Domain lead
Evidence preparer, not workflow owner. Makes packets the formal gate can rule on.
Stage operative
The agent that issues a binary verdict for a single SDLC stage.
Spec creator
Pre-implementation agents (Change Impact, Planner, Spec Guardian, Architect, DB Reviewer).
ADR
Architecture Decision Record. Constraints that bind future work. Lives in decisions/.
Coordination file
A file multiple lanes might touch (manifests, lockfiles, base configs). Always declared in the packet.
Evidence packet
Required closing block of every implementation response — what changed, what was tested, next packet.
Transplant
The act of dropping the kit into a new repo and running the bootstrap checklist to regrow specificity.

Mental model in one paragraph

A typed spec defines the truth. A control-plane orchestrator reads the spec and the live state file, decides which narrow agent acts next, and routes the work through stages. Each stage agent issues a hard verdict; domain leads prepare the evidence those verdicts depend on. Lanes keep parallel work parallel. State files keep yesterday's truth available. A learning loop turns recurring lessons into durable rules — but never silently, never without a human-approved PR. The whole kit is portable: drop it in, profile the environment, and the same discipline regrows in the new repo.

Spec SDLC is a portable, lean transplant of a mature spec-first delivery kit. Read those directories alongside this document — and read SOURCE_MAP.md, COMPONENT_MANIFEST.md, ASSEMBLY_REPORT.md, and SWOT_ANALYSIS.md before customising — and the moving parts will line up with the named roles you saw here.
